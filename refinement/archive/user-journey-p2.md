## UJ21 — Lab/quiz model labels are bare debug strings (A7)

> *"`LabRunner.svelte:48-50`: renders raw `{lab.model}` (e.g. `glm-5.2`) with no framing. Quiz runner similar. Reads as developer output, not product copy."*

**Mechanism (decided):**
- Frame the raw model id as product copy. Replace the bare `{lab.model}` / `{quizzesStore.current.model}` line with a small **"Generated by `<provider>` · `<model>`"** chip, consistent with the UJ3 chat composer chip.
- The lab/quiz rows store `model` (the `config.defaultModel` at save time — `labs.svelte.ts:115`, `quizzesStore` similar). They do **not** store the provider display name. Two options:
  1. Render `Generated by <model>` only (model id, no provider) — accurate to the stored data, no new lookup.
  2. Resolve the provider name from the stored model id (fragile — model ids aren't unique across providers).

**Decided:** option 1, but soften the model id into a readable label. Render: *"Generated by `<model>`"* in muted small text, matching the chip styling from UJ3. This is honest (the row stores the model used at generation time, which may differ from the current active provider — a feature, not a bug: it tells the user *which model produced this artifact*). Pair visually with UJ3's chip so the app has one consistent "model attribution" pattern.

**Files modified**
- `src/lib/components/labs/LabRunner.svelte` (`:47-51`) — replace
  ```svelte
  {#if lab.model}
    <p class="text-xs text-muted-foreground">{lab.model}</p>
  {/if}
  ```
  with a framed chip:
  ```svelte
  {#if lab.model}
    <p class="text-xs text-muted-foreground">Generated by {lab.model}</p>
  {/if}
  ```
  (Keep it text, not a heavy chip — the lab/quiz header is already compact.)
- `src/lib/components/quizzes/QuizRunner.svelte` (`:48-50`) — symmetric change: `Generated by {quizzesStore.current.model}`.

**Tests**
- None automated (string change).

**Manual gate**
- `/lab/[id]` and `/quiz/[id]`: the model line reads "Generated by glm-5.2" (or whichever), no longer a bare debug token. Consistent with the chat composer chip.

### UJ21 — decisions / open items
- **RESOLVED:** frame as "Generated by `<model>`" (option 1); don't reverse-resolve provider name from model id (fragile).

## UJ23 — Two-step MCQ + numbered quiz rail (B10)

> *[!NOTE on B10]: "two-step mcq and numbered rail for longer quizzes."*

Two changes, one phase (both are quiz-runner UX):

### UJ23a — Two-step MCQ (select → confirm)

**Mechanism (decided):**
- Today `choose(i)` fires `onAnswer(i)` immediately on click (`McqQuestion.svelte:31-35`) — a misclick is permanent. Change to a two-step flow: click selects (optimistic highlight), a **"Submit answer"** button confirms. The radio still records `localPick`; the persist only fires on submit.
- Once answered (the `answer` row exists), the flow is locked as today.

**Files modified**
- `src/lib/components/quizzes/McqQuestion.svelte`
  - `choose(i)` (`:31-35`) → set `localPick = i` only; do **not** call `onAnswer`.
  - Add a **"Submit answer"** button (disabled until `localPick !== null` and `!locked`) that calls `onAnswer(localPick)`.
  - The `selected`/`optionClass` logic (`:29,37-49`) already keys off `localPick` for the optimistic pick and `answer` for the locked state — no change needed there; the highlight now means "selected, not yet submitted."
  - Edge: if the user navigates away with a `localPick` but no submit, nothing is persisted (correct — no half-answer).

### UJ23b — Numbered question rail for longer quizzes

**Mechanism (decided):**
- Add a numbered progress rail to the quiz runner: a row of numbered chips (1…N), each showing answered/unanswered/current state, clickable to jump.
- Show only when the quiz has ≥ some threshold of questions (e.g. ≥5) to avoid clutter on short quizzes. **Decided threshold: ≥5.**

**Files modified**
- `src/lib/components/quizzes/QuizRunner.svelte`
  - Above the question list (`:76`), add a numbered rail (only when `quizzesStore.total >= 5`):
    ```svelte
    {#if quizzesStore.total >= 5}
      <div class="flex flex-wrap gap-1">
        {#each quizzesStore.questions as q, ord (q.id)}
          {@const answered = !!quizzesStore.answers[q.id]}
          <button class="quiz-nav-chip {answered ? 'answered' : ''}" onclick={() => jumpTo(ord)}>
            {ord + 1}
          </button>
        {/each}
      </div>
    {/if}
    ```
  - `jumpTo(ord)`: scroll the question into view (`document.getElementById('quiz-q-' + qId)?.scrollIntoView({behavior:'smooth', block:'center'})`). Add `id="quiz-q-{q.id}"` to each question `<li>` (`:79`).
  - CSS for `.quiz-nav-chip`: small square, muted; `.answered` gets an emerald tint; current question (track via scroll position or a `currentOrd` state) gets a ring.

**Tests**
- None automated (DOM/scroll). The `answerMcq` store path is unchanged (still fires on submit; just later).

**Manual gate**
- A 3-question quiz: no rail (below threshold); MCQ is two-step (select → Submit). A 10-question quiz: numbered rail appears, answered chips turn emerald, click a chip → jumps to that question. Misclick no longer locks (can change selection before Submit).

### UJ23 — decisions / open items
- **RESOLVED (UJ23a):** two-step MCQ (select → Submit button); persist on submit only.
- **RESOLVED (UJ23b):** numbered rail when `total >= 5`; click to jump via `scrollIntoView`.

## UJ26 — Route-level code splitting (C4)

> *"No `+page.ts` load functions exist; all data loading is in `onMount`. SvelteKit's `sveltekit-preload-data="hover"` prefetches on hover, but there's no lazy route chunking — the initial bundle carries every page."*

**Mechanism (decided):**
- Add `+page.ts` (and `+layout.ts` where useful) `load` functions that do the data fetching currently in `onMount`, so SvelteKit can code-split per route and prefetch.
- This is lower priority for a local app (the audit rates it P2/M) but helps first-paint on the browser runtime.

**Decision surfaced while planning — SPA + adapter-static constraints**

The app is a **static SPA** (`@sveltejs/adapter-static`, no SSR — `AGENTS.md`). `load` runs client-side only; it can't do server work. The win is purely route chunking + prefetch, not server rendering. **Decided:**
- Migrate the `onMount` data loads to `+page.ts` `load` functions where the data is needed before render (chat list, lab list, quiz list, search). Keep it client-side (`export const ssr = false` is implicit from the SPA setup).
- Don't over-invest: this is P2. Prioritize the routes that carry heavy dependencies (the markdown/KaTeX/mermaid stack) so they chunk separately from the initial bundle. The chat route is already heavy; splitting it out helps first-paint on `/`.

**Files modified**
- `src/routes/chat/+page.ts` **(new)** — `load` returns `{ roots: await listRootChats() }`; the `+page.svelte` reads `data.roots` instead of fetching in `onMount`.
- `src/routes/chat/[id]/+page.ts` **(new)** — `load` returns `{ chatId: params.id }` (the heavy `chatStore.load` + nav load stays in the component, since it depends on the store singleton; just enable route chunking by having a `+page.ts`).
- `src/routes/lab/+page.ts`, `src/routes/quiz/+page.ts`, `src/routes/search/+page.ts` — similar thin `load` functions to enable chunking + prefetch.
- The corresponding `+page.svelte` files — read from `data` instead of `onMount` where migrated.
- This is intentionally light; full data-loader migration is a larger refactor not warranted at P2.

**Tests**
- None automated (build output). Verify chunking via `pnpm build` + inspecting `build/` for per-route chunks.

**Manual gate**
- `pnpm build` → the initial bundle no longer carries every route (verify chunk sizes). `pnpm dev` → hover a sidebar link → the route prefetches (Network tab). First-paint on `/` is no worse (ideally better) than before.

### UJ26 — decisions / open items
- **RESOLVED:** thin client-side `+page.ts` `load` functions for route chunking + prefetch (SPA, no SSR). Keep store-dependent loads in-component; migrate list-route fetches to `load`. Light touch (P2).
