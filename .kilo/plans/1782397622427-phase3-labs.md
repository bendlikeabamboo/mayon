# P3 — Labs

Generate a hands-on lab from any chat and run it with a persisted checklist.
Derived from `refinement/architecture.md` §7 and `refinement/phased-plan.md` (P3).
Builds on the P0–P2 foundation; no schema migration required.

## Resolved decisions

| Area | Decision |
| --- | --- |
| Structured output | Prompt + fenced-JSON parse. Uniform across all providers; reuses existing SSE streaming path. No per-adapter wire changes. |
| Storage | Map generated `{ title, intro, steps[] }` → existing `labs.content` (markdown body); `checklist[]` → existing `labs.checklist` JSON column. **No migration.** |
| Generation UX | Background spinner on the chat page; navigate to `/lab/<id>` on completion. |
| Editable prompt | Default prompt constant + `labPrompt` settings KV override (Settings UI). |
| Routing | `/lab` (list, grouped by chat) + `/lab/[id]` (runner). "Generate lab" button + existing-labs chips on `/chat/[id]`. |
| Malformed JSON | Retry up to 2×, then formatted error + offer to save raw text as a lab anyway. |
| Context | Reuse `assembleContext(chatId)` (ancestors + excerpt) — same input the chat stream uses. |
| Validation | Add **Zod** dependency for schema validation (reused by P4 quiz payloads). |

## Out of scope

- Runnable lab sandbox, spaced repetition, regenerating labs beyond checklist toggles.
- Editing lab `content` in-app (generated markdown is read-only; only checklist is interactive).
- P4 (quizzes/grading) — independent phase.

## Affected boundaries & data flow

```
/chat/[id]  ──"Generate lab"──▶  labsStore.generate(chatId)
                                     │
                assembleContext(chatId)  ◀── reuse (ancestors + excerpt)
                getActiveProvider()     ◀── reuse (P1)
                                     │
                Provider.generateLab(ctx, { labPrompt? })   ◀── NEW on interface
                  │  = prompt + stream tokens + fenced-JSON extract
                  │  + Zod validate (retry ≤2 on failure)
                  ▼
                labsRepo.create({ chatId, title, content, checklist, model })
                  ▼
                goto(`/lab/${lab.id}`)
                                     │
/lab/[id]  ◀── LabRunner ── renders markdown content + interactive checklist
                       checklist toggles ──▶ labsRepo.toggleChecklistItem (optimistic)
```

**Boundary check (per AGENTS.md):** components/stores call repositories + the AI client seam only. `generateLab` lives behind `Provider`/`getActiveProvider()`. No route imports `db` directly.

## Tasks (ordered)

### 1. Add Zod dependency
- `pnpm add zod`.
- Verify it imports cleanly under the TS config (`src/lib/ai/generate/` will use it).

### 2. Lab payload schema + parser — `src/lib/ai/generate/lab.ts` (NEW)
- Define the generated lab shape and a Zod schema:
  ```ts
  export interface GeneratedLab {
    title: string;
    intro: string;          // markdown
    steps: string[];        // markdown, ordered
    checklist: { text: string }[];   // ids assigned on persist
  }
  export const GeneratedLabSchema: z.ZodType<GeneratedLab>;  // strict
  ```
- `extractFencedJson(raw: string): string | null` — pull the first ```json fenced block (regex); fall back to the whole string if no fence (some models emit bare JSON).
- `parseGeneratedLab(raw: string): GeneratedLab` — `JSON.parse` + `GeneratedLabSchema.parse`; throw `LabParseError` (new typed error in this file, not in `errors.ts` — it's internal, not a transport error) on any failure.
- `toLabContent(lab: GeneratedLab): { title: string; content: string; checklist: LabChecklistItem[] }` — flatten `intro` + numbered `steps` into a single markdown body; assign `uuid()` to each checklist item.
- Unit tests (`lab.test.ts`): valid fenced, bare JSON, missing fields, extra fields (strict reject), non-JSON text, `toLabContent` shape.

### 3. Lab generation orchestrator — `src/lib/ai/generate/generate.ts` (NEW)
- `DEFAULT_LAB_PROMPT` constant (system message instructing the model to emit the exact JSON shape inside a ```json fence).
- `readLabPrompt(): Promise<string>` — read `labPrompt` from settings KV, fall back to `DEFAULT_LAB_PROMPT` (mirrors `getActiveProvider`'s settings-read pattern).
- `generateLab(ctx: ChatMessage[], opts?: { signal?: AbortSignal }): Promise<GeneratedLab>`:
  1. Accumulate `provider.chatStream([{role:'system', content: labPrompt}, ...ctx], { signal })` into a string (reuse the same token-accumulation loop as `chatStore.send`).
  2. `parseGeneratedLab(raw)`.
  3. On failure, retry up to **2×** total: append `{role:'assistant', content: raw}` + `{role:'user', content: 'That was not valid JSON matching the schema. Output ONLY a ```json block with {title,intro,steps[],checklist[]}.'}` then re-stream.
  4. After max attempts, throw a `LabGenerationError` carrying the last raw text (caller decides whether to save-as-raw).
- Unit tests with a stub provider (`adapters/stubs.ts` already exists; add a controllable stub emitting fixture JSON / garbage).

## Validation plan (manual acceptance gates)

Mirroring the AGENTS.md gate style:

- **Generate (browser):** `pnpm dev` → open `/chat/<id>` with a real conversation → click **Generate lab** → spinner → navigates to `/lab/<id>` → lab title, markdown steps, and checklist all render → tick checklist items → **reload** → ticks persist (proving `labs` persistence via OPFS).
- **Generate (desktop):** `pnpm tauri dev` → same flow → lab + checklist survive an **app restart**.
- **Reachability:** `/lab` lists the lab grouped under its chat; "Back to chat" returns to `/chat/<id]`; the chat page shows an existing-lab chip linking back.
- **Prompt override:** set a custom `labPrompt` in Settings → regenerate → output reflects the customized instruction; reset → reverts to default.
- **Malformed-output fallback:** point generation at a weak/local model or inject a fixture → after retries the UI offers "save raw"; accepting it creates a lab with empty checklist.
- **No active provider:** trigger generate with no provider set → formatted "Missing API key / no active provider" error in the chat error block (reuses `formatProviderError`), never a crash.

> Desktop build needs the GTK/WebKit dev libs (AGENTS.md); verify it on a real machine, not headless CI.

## Risks / notes

- **Strict-Zod retries can burn tokens.** Mitigation: cap at 2 retries; the prompt explicitly demands the fence; the fallback saves raw so nothing is lost.
- **`labs.content` as a single markdown blob** means no per-step DB querying — acceptable (labs are leaf artifacts, rendered whole). If step-level state is ever needed, add a `payload` JSON column later (no break to `content`).
- **Cross-provider consistency** relies entirely on prompt compliance. The default prompt must be explicit about the fence and field names; document that weaker models (small Ollama) may hit the retry/fallback path more often.
- **Abort** must cancel the accumulation loop cleanly (propagate `AbortError` and swallow it in the store, like `chatStore.send`).
