# LS1 — Strategy core (registry + note + dropdown)

Implementation plan for the **LS1** phase of
`refinement/learning-structure-phased.md`. The authoritative design is
`refinement/learning-structure.md` (treat §6 for block text, §7 for the note
shape). This plan is the executable breakdown for an implementation agent.

## Goal

A selectable teaching **strategy** reaches the system prompt for all three modes
(`socratic` / `explainer` / `build`) via the existing
`assembleContext → buildBriefSystemNote` chokepoint, replacing the one-line
`MODE_INSTRUCTIONS`. After LS1 the three core complaints (Explainer
structure/density, Socratic substance floor, Build code emphasis) are addressed
at the prompt level; pacing gates work as plain text ("type continue"). No
admonition renderer (LS2) and no suggested-reply chips (LS3) in this phase.

## Locked decisions

1. **Always-emit the strategy block.** Every *parseable* brief emits its block
   (`brief.scopeStrategy` if valid, else `defaultStrategyFor(mode)`).
   `MODE_INSTRUCTIONS` is **deleted entirely**. Legacy briefs (no
   `scopeStrategy`) resolve to the mode-default block — output gets richer, as
   intended by design §7. The **null-brief** path is the only escape hatch and
   stays byte-for-byte identical.
2. **Snapshot resolution.** `applyProfile` resolves `scopeStrategy`
   (`brief > profile > mode-default`) and bakes it into the stored brief,
   exactly as it already does for `level`/`mode`. `buildBriefSystemNote` is
   therefore **profile-less** (it only ever sees a parsed `LearningBrief` —
   `context.ts:114` has no profile, by design).
3. **Three default strategies only** in LS1: `guided-curriculum` (Explainer),
   `guided-inquiry` (Socratic), `workshop` (Build). The other 7 ids are
   declared in the `ScopeStrategyId` union but have no registry entry yet (LS4).
4. **Block text verbatim** from design §6.1–6.3 (including the `> [!NOTE]` /
   `> [!WARNING]` examples in `workshop`). In LS1 those render as plain
   blockquotes — this is the intentional short-lived degraded state fixed by LS2.
5. **No DB change.** Strategy rides in the existing `chats.brief` JSON column →
   no migration, no `db:generate`, no `bundle:migrations`.
6. **Out of scope:** admonition renderer (LS2), composer chips (LS3), profile
   default UI (LS4), inferred-brief `scopeStrategy` (LS4), `summarizeBrief`
   change (leave as-is — `goal · level · mode` chip is unchanged).

## Architecture fit (verified from current code)

- `MODE_INSTRUCTIONS` + its phrases (`brief.ts:61-65,176`) are referenced only
  in `brief.ts` and one test (`brief.test.ts:111-113`). Clean blast radius.
- `context.ts:112-114` → `parseBrief(root.brief)` → `buildBriefSystemNote(brief)`
  is the single integration point; no change needed in `context.ts`.
- `generate-brief.ts:35` `GeneratedBrief = Pick<LearningBrief,…>` is an explicit
  pick, so widening `LearningBrief` does **not** widen the inferred schema.
- Circular-import safety: `strategies.ts` imports only `type { BriefMode }`
  (erased); `brief.ts` imports runtime exports from `strategies.ts`. No runtime
  cycle.

## Tasks (ordered)

### 1. New module `src/lib/chat/strategies.ts`

Exports (signatures):

```ts
import type { BriefMode } from './brief';

export type ScopeStrategyId =
  | 'guided-curriculum' | 'deep-dive' | 'quick-orientation' | 'reference-manual'
  | 'guided-inquiry' | 'devils-advocate' | 'case-based'
  | 'workshop' | 'tutorial' | 'pair-programming';

export interface ScopeStrategy {
  id: ScopeStrategyId;
  label: string;   // dropdown text, e.g. "Guided curriculum"
  hint: string;    // one-line "what you get"
  modes: BriefMode[];
  gated: boolean;  // ends turns with a pacing gate (drives chips in LS3)
  block: string;   // the prompt payload (design §6)
}

export const SCOPE_STRATEGIES: ScopeStrategy[]; // 3 entries in LS1
export const SCOPE_STRATEGY_IDS: readonly ScopeStrategyId[]; // all 10 ids

export function isScopeStrategyId(v: unknown): v is ScopeStrategyId;
export function strategiesForMode(m: BriefMode): ScopeStrategy[];
export function defaultStrategyFor(m: BriefMode): ScopeStrategyId;
export function resolveStrategy(brief: Partial<LearningBrief>, profile: LearnerProfile): ScopeStrategy;
```

- Define the 3 registry entries with `block` text copied **verbatim** from
  design §6.1 (`guided-curriculum`), §6.2 (`guided-inquiry`), §6.3 (`workshop`).
- `gated`: `guided-curriculum` = true, `guided-inquiry` = false, `workshop` = true.
- `modes`: each strategy's single mode.
- `defaultStrategyFor`: explainer→`guided-curriculum`, socratic→`guided-inquiry`,
  build→`workshop`.
- `resolveStrategy`: id = `brief.scopeStrategy ?? profile.scopeStrategy ??
  defaultStrategyFor(brief.mode ?? DEFAULT_MODE)`; return the registry entry (or
  the mode-default entry if the id isn't registered yet, e.g. an LS4 id). Never
  throws — falls back to `defaultStrategyFor` for any unknown/unregistered id.
- Import `LearningBrief`, `LearnerProfile`, `DEFAULT_MODE` as needed (these come
  from `brief.ts`; `LearningBrief`/`LearnerProfile` as types, `DEFAULT_MODE` as a
  runtime value — acceptable; no runtime cycle since `brief.ts` will import from
  `strategies.ts` only the registry/helpers).

### 2. Extend `src/lib/chat/brief.ts`

- **Types:** add `scopeStrategy?: ScopeStrategyId` to `LearningBrief`, to
  `LearnerProfile`, and to `ResolvedBriefFields` (id, always resolved).
- **`DEFAULT_PROFILE`:** leave `scopeStrategy` unset (→ mode-default at resolve
  time); do NOT hardcode a value (keeps the mode→default mapping single-sourced).
- **`parseBrief`:** add `if (isScopeStrategyId(obj.scopeStrategy))
  brief.scopeStrategy = obj.scopeStrategy;` (one guard line, imported from
  `strategies.ts`). Unknown ids are dropped → mode-default later (total parser
  invariant preserved).
- **`applyProfile`:** resolve `scopeStrategy` via
  `resolveStrategy(brief, profile).id` and add it to the returned
  `ResolvedBriefFields`. Precedence is handled inside `resolveStrategy`.
- **Delete `MODE_INSTRUCTIONS`** and rewrite `buildBriefSystemNote`:
  - Resolve the strategy from the brief alone (no profile):
    `const strat = resolveStrategyFromBrief(brief)` where that helper does
    `SCOPE_STRATEGIES.find(s => s.id === brief.scopeStrategy) ??
    by-id(defaultStrategyFor(mode))`. (Or inline; keep it in `strategies.ts` as
    a small helper, e.g. `strategyForBrief(brief: LearningBrief): ScopeStrategy`
    — recommended to keep `brief.ts` free of registry lookups.)
  - New note shape (design §7), keeping the calibration line and goal line,
    appending the block:
    ```
    You are a personal learning tutor. Calibrate to this learner's brief:
    - Goal: <goal>
    - Level: <level>  · Context: <context>  · Mode: <mode>  · Scope: <scope>
    - Structure: <strategy.label>  (unless scope overrides the budget)

    <strategy.block>

    Teach to the goal at the stated level; stay within scope.
    When the learner can do the goal, say so.
    ```
  - **scope budget line:** if `brief.scope` is present, add one line tightening
    density/units (e.g. `"The learner set this budget: <scope>. Honor it when
    it tightens density or unit count below the structure's defaults."`); if
    empty, omit (the strategy's defaults stand). Keep this minimal and optional.
- **Re-export** from `brief.ts`: `ScopeStrategyId`, `ScopeStrategy`,
  `SCOPE_STRATEGIES`, `strategiesForMode`, `defaultStrategyFor`,
  `resolveStrategy`, `strategyForBrief` (so existing importers of brief types
  keep working).
- **Do not** touch `summarizeBrief` (LS1 leaves the chip as-is).

### 3. Extend `src/lib/components/chat/BriefCard.svelte`

- Add a **Structure** `<select>` below the Mode select. Options =
  `strategiesForMode(modeVal)` (re-derives reactively when `modeVal` changes —
  if the current strategy id isn't in the new mode's list, snap to
  `defaultStrategyFor(modeVal)`).
- Local state: `let scopeStrategy = $state(untrack(() =>

- In `buildBrief()`, always write `b.scopeStrategy = scopeStrategy` (snapshot the

- Hint text under the select = the active strategy's `hint`.

## Acceptance

- `pnpm test`, `pnpm check`, `pnpm lint` all clean.
- **Manual (browser):** `pnpm dev` → new chat intake → pick a structure (e.g.
  Explainer + Guided curriculum) → "Start learning" → first reply is an
  orientation (advance organizer + ToC + "continue?" gate). Switch mode → the
  Structure dropdown re-derives. Edit via the summary chip → change strategy →
  recalibrates on the next turn. **Reload persists.** A "Just start chatting"
  (null-brief) chat behaves exactly as today (no system note).
- **Socratic:** a reply follows anchor → framing → single probe; never answers
  its own probe; ≥~120 words.
- **Build:** a reply is code-first with a `> [!NOTE]`/`> [!WARNING]` that
  renders as a plain blockquote in LS1 (the known degraded state).
- **Branching:** branch a root mid-curriculum → the child inherits the strategy
  (read via `rootId`); its turns follow the same structure.

## Risks / edge cases

- **Behavior change for ALL existing chats:** every previously-created brief now
  resolves a mode-default block instead of the one-liner. This is intended
  (design §7) but means the saved "mode-specific teaching guidance" test must be
  rewritten, not just appended.
- **`resolveStrategy` must never throw** on a valid-but-unregistered id (LS4 ids)
  or garbage — fall back to `defaultStrategyFor(mode)`. `parseBrief` already
  drops garbage, but defense-in-depth in the resolver keeps the note builder safe.
- **Mode switch resets strategy:** when the user changes Mode in the card, the
  current strategy id may no longer belong to that mode; snap to
  `defaultStrategyFor(newMode)` (don't leave a stale id in the stored brief).
- **Token cost:** block (~200–400 tokens) added once per assembled context —
  negligible, consistent with today's note.
- **No streaming/post-processing** in LS1: density is instructed, never enforced
  by rewriting tokens.
