# UX4 — FTS5 full-text search (round 1)

**Epic:** `refinement/ui-ux-phased.md` → UX4 (refinement #6, round 1).
**Status:** Execution-ready. All decisions resolved; three are **corrections to the
spec**, surfaced by code inspection (same posture as the UX3 plan).

## Goal

A `/search` route plus a header/sidebar entry point that lets a learner find any
message, chat title, lab, or quiz prompt across their local DB, fully offline,
ranked by relevance (bm25), with snippet + highlight, and deep-links that jump
straight to the hit. Powered by an **FTS5** index kept in sync by **triggers** so
every write path (including UX2's `deleteBranch` cascade) stays consistent
without app-layer indexing.

UX2 + UX3 are already shipped (verified: `chats.deleteBranch`, the
`snapshot`/`restore`/`dispose` seam, `rebootstrapWith`, and `DataSection.svelte`
all exist). UX4 reuses the **reserved "Rebuild search index" slot** in
`DataSection.svelte:98`.

## Decisions (resolved)

- **❗ Correction 1 — deep-link reuses the EXISTING hash mechanism, not `?m=`.**
  The spec (decision H) chose `/chat/[id]?m=<messageId>` over a hash. **Code
  inspection contradicts this:** `src/routes/chat/[id]/+page.svelte:226`
  (`handleHashScroll`) already reads `window.location.hash` → `m` param →
  `scrollIntoView` to `#msg-{id}` with a `requestAnimationFrame` retry that
  solves the exact render-race the spec cited as the reason to avoid hashes. It
  is already used by the **expound** feature (`#m=…&b=…`). Adding a *second*
  `?m=` query-param reader would be inconsistent with expound and duplicate
  solved logic. **Search results therefore link to `/chat/[id]#m=<messageId>`**
  and **require zero changes to the chat route** for the core scroll. (A brief
  highlight flash is an optional add — see UI.) **Net: decision H is overridden
  by reality; reusing the hash is lower-risk and consistent.**

- **❗ Correction 2 — the FTS5 table is REGULAR (text-storing), NOT contentless
  (`content=''`).** The spec's DDL says `content=''` (contentless) AND wants
  `snippet()`/`highlight()`. Those two are incompatible: a contentless FTS5
  table cannot serve `snippet()`/`highlight()` (it stores no original text), and
  a single contentless/external-content FTS5 table can map to only **one** source
  table whereas we unify **four** (messages, chats, labs, quiz_questions).
  **Resolution:** a normal FTS5 table that stores the text (duplicating it). At
  a single-learner's scale this is a few MB and is the price of correct,
  multi-source snippets + ranking. `snippet()`/`highlight()` then work directly.

- **❗ Correction 3 — triggers index RAW text; `stripIndexNoise` lives in JS and
  is applied by `rebuildIndex()` (and its unit test), NOT in the triggers.**
  SQLite has no built-in regex, so the mermaid/KaTeX stripping the spec wanted
  in the trigger path is not expressible in portable SQL. The migration's
  one-time backfill therefore inserts **raw** text (matching exactly what the
  live triggers do → the index is always self-consistent). `stripIndexNoise(md)`
  remains a pure, unit-tested helper used by `rebuildIndex()` so a user who
  clicks "Rebuild search index" gets the cleaner (mermaid/KaTeX-stripped)
  snippet quality. Trade-off: brand-new rows may show a stray mermaid/katex token
  in a snippet until the next rebuild — cosmetic only; search correctness is
  unaffected.

- **Routing keys generalised.** The spec named a single `message_id UNINDEXED`.
  Four source kinds need different identifiers, so the schema uses
  `chat_id UNINDEXED` + `ref_id UNINDEXED` (message id / chat id / lab id /
  question id) + `quiz_id UNINDEXED` (populated only for `quiz_question`, for the
  `/quiz/{quiz_id}` deep-link). `chat_title` + `root_id` for tree-grouping are
  fetched by a `LEFT JOIN chats` at query time (cheap; result sets are small).

- **Indexed kinds (v1):** `message`, `chat`, `lab`, `quiz_question`.
  `branch_sources` is **excluded** — its `excerpt` is already indexed transitively
  as part of the source message's content; indexing it separately would
  duplicate. (Noted as out-of-scope; easy to add later.)

- **Tokenizer:** `unicode61 remove_diacritics 2` (spec decision F). `trigram`
  deferred (needs ≥ 3.34 + triples index size).

- **MATCH query construction:** `buildMatchQuery(raw)` splits on whitespace,
  drops empty tokens, wraps each in double quotes (`"tok"`), joins with a space
  → implicit **AND** phrase-token search. Quoting makes FTS5 treat specials
  literally → no escaping hazards, no unicode stripping needed. Prefix
  matching (`*`) is deferred (noted). Unit-tested.

- **Snippet rendering is injection-safe (no `{@html}`).** `snippet()` is called
  with markers `char(1)`/`char(2)` (`\x01`/`\x02`, never in real text); the
  `/search` route splits the returned string on those and renders Svelte `<mark>`
  elements.

- **Hand-authored migration** (spec decision E): `drizzle/0006_search_fts.sql`
  + a journal entry; `pnpm bundle:migrations` regenerates
  `src/lib/db/driver/migrations.ts`. **Not** added to `schema.ts` (FTS5 virtual
  tables aren't drizzle models); the migrator just runs raw SQL.

## Migration — `drizzle/0006_search_fts.sql` (new, hand-authored)

**Workflow:** author the `.sql` with `--> statement-breakpoint` markers between
each *complete* statement (a trigger including its `BEGIN…END;` is ONE chunk —
the bundle script splits only on the breakpoint marker, so internal trigger
`;` survive as one statement, which SQLite parses correctly); append the journal
entry; run `pnpm bundle:migrations` (it computes the sha256 `hash` itself).

`drizzle/meta/_journal.json` — append:
```json
{ "idx": 6, "version": "6", "when": <NOW_MS, >1783008654126>, "tag": "0006_search_fts", "breakpoints": true }
```

DDL (each `--> statement-breakpoint`-separated):

```sql
-- content-storing FTS5 (see Correction 2). title/body indexed; the rest UNINDEXED.
CREATE VIRTUAL TABLE search_fts USING fts5(
  kind UNINDEXED,
  title,
  body,
  chat_id UNINDEXED,
  ref_id UNINDEXED,
  quiz_id UNINDEXED,
  tokenize = 'unicode61 remove_diacritics 2'
);
--> statement-breakpoint
-- messages ──────────────────────────────────────────────────────────
CREATE TRIGGER search_fts_messages_ai AFTER INSERT ON messages BEGIN
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    VALUES('message','',new.content,new.chat_id,new.id,NULL);
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_messages_ad AFTER DELETE ON messages BEGIN
  DELETE FROM search_fts WHERE kind='message' AND ref_id = old.id;
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_messages_au AFTER UPDATE ON messages BEGIN
  DELETE FROM search_fts WHERE kind='message' AND ref_id = old.id;
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    VALUES('message','',new.content,new.chat_id,new.id,NULL);
END;
--> statement-breakpoint
-- chats (title only) ────────────────────────────────────────────────
CREATE TRIGGER search_fts_chats_ai AFTER INSERT ON chats BEGIN
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    VALUES('chat',new.title,'',new.id,new.id,NULL);
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_chats_ad AFTER DELETE ON chats BEGIN
  DELETE FROM search_fts WHERE kind='chat' AND ref_id = old.id;
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_chats_au AFTER UPDATE ON chats BEGIN
  DELETE FROM search_fts WHERE kind='chat' AND ref_id = old.id;
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    VALUES('chat',new.title,'',new.id,new.id,NULL);
END;
--> statement-breakpoint
-- labs (title + content) ────────────────────────────────────────────
CREATE TRIGGER search_fts_labs_ai AFTER INSERT ON labs BEGIN
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    VALUES('lab',new.title,new.content,new.chat_id,new.id,NULL);
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_labs_ad AFTER DELETE ON labs BEGIN
  DELETE FROM search_fts WHERE kind='lab' AND ref_id = old.id;
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_labs_au AFTER UPDATE ON labs BEGIN
  DELETE FROM search_fts WHERE kind='lab' AND ref_id = old.id;
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    VALUES('lab',new.title,new.content,new.chat_id,new.id,NULL);
END;
--> statement-breakpoint
-- quiz_questions (prompt; chat_id resolved via the quiz) ────────────
CREATE TRIGGER search_fts_qq_ai AFTER INSERT ON quiz_questions BEGIN
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    SELECT 'quiz_question','',new.prompt,q.chat_id,new.id,new.quiz_id
    FROM quizzes q WHERE q.id = new.quiz_id;
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_qq_ad AFTER DELETE ON quiz_questions BEGIN
  DELETE FROM search_fts WHERE kind='quiz_question' AND ref_id = old.id;
END;
--> statement-breakpoint
CREATE TRIGGER search_fts_qq_au AFTER UPDATE ON quiz_questions BEGIN
  DELETE FROM search_fts WHERE kind='quiz_question' AND ref_id = old.id;
  INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
    SELECT 'quiz_question','',new.prompt,q.chat_id,new.id,new.quiz_id
    FROM quizzes q WHERE q.id = new.quiz_id;
END;
--> statement-breakpoint
-- one-time backfill (RAW text — see Correction 3) ───────────────────
INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
  SELECT 'chat', title, '', id, id, NULL FROM chats;
--> statement-breakpoint
INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
  SELECT 'message', '', content, chat_id, id, NULL FROM messages;
--> statement-breakpoint
INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
  SELECT 'lab', title, content, chat_id, id, NULL FROM labs;
--> statement-breakpoint
INSERT INTO search_fts(kind,title,body,chat_id,ref_id,quiz_id)
  SELECT 'quiz_question', '', qq.prompt, q.chat_id, qq.id, qq.quiz_id
  FROM quiz_questions qq JOIN quizzes q ON q.id = qq.quiz_id;
```

> A `DELETE FROM search_fts WHERE kind=? AND ref_id=?` is a valid FTS5 statement
> (FTS5 supports arbitrary-`WHERE` DELETE; UNINDEXED cols force a linear scan,
> which is fine at single-learner scale). The `UPDATE` triggers do delete-then-
> insert (FTS5 has no partial row update). `chats.touch()` re-indexing the title
> row on every `updatedAt` bump is negligible overhead, accepted.

> Do **not** add `search_fts` to `schema.ts`. Leave a header comment in the
> `.sql` stating it is intentionally not a drizzle model and is sync'd solely by
> these triggers.

## Repository — `src/lib/db/repositories/search.ts` (new)

Mirrors the existing repo style (`awaitDb()` for drizzle, `getDriver()` for raw
SQL — FTS queries use `getDriver().query` since they reference FTS5 aux
functions drizzle can't express).

**Exports:**
- `type SearchKind = 'message' | 'chat' | 'lab' | 'quiz_question';`
- `interface SearchHit` — `{ kind, chatId, refId, quizId, title, chatTitle, rootId, snippetTitle, snippetBody, rank }`
  (`snippet*` carry `\x01`/`\x02` markers for the UI to split).
- `stripIndexNoise(md: string): string` — pure helper: strip ```` ```mermaid ``` ````
  fenced blocks and KaTeX (`$…$` / `$$…$$`) source; **keep** inline code and
  non-mermaid fenced code. (Unit-tested per spec decision G.)
- `buildMatchQuery(raw: string): string | null` — pure helper (see Decisions).
- `fts5Available(): Promise<boolean>` — `getDriver()` try `CREATE VIRTUAL TABLE
  _fts_probe USING fts5(x); …; DROP TABLE _fts_probe;` → bool. Used by the UI to
  decide whether to show search (gate-time/runtime degrade) and by tests.
- `searchRepo.search(query, opts?: { limit?: number; kinds?: SearchKind[] }):
  Promise<SearchHit[]>` — builds the MATCH string (returns `[]` on null/empty),
  runs the bm25 query (see SQL below), maps rows.
- `searchRepo.rebuildIndex(): Promise<void>` — `DELETE FROM search_fts;` then
  re-insert each source kind through the driver in chunked batches, applying
  `stripIndexNoise` to bodies (the clean-index path, Correction 3). Idempotent.
- `searchRepo.stripIndexNoise`, `searchRepo.buildMatchQuery`, `searchRepo.fts5Available`
  also exposed for tests/UI.

**Search SQL** (one round-trip; `?1` = MATCH expr, `?2` = limit, optional
`?3..` = kinds via `AND search_fts.kind IN (?,…)`, or omitted):
```sql
SELECT
  search_fts.kind         AS kind,
  search_fts.chat_id      AS chat_id,
  search_fts.ref_id       AS ref_id,
  search_fts.quiz_id      AS quiz_id,
  search_fts.title        AS title,
  c.title                 AS chat_title,
  c.root_id               AS root_id,
  snippet(search_fts, 2, char(1), char(2), '…', 12) AS snippet_body,
  snippet(search_fts, 1, char(1), char(2), '…', 8)  AS snippet_title,
  bm25(search_fts)        AS rank
FROM search_fts
LEFT JOIN chats c ON c.id = search_fts.chat_id
WHERE search_fts MATCH ?1
ORDER BY rank
LIMIT ?2;
```
Column indices for `snippet`: `0=kind, 1=title, 2=body`. `bm25()` is
negative-ranked (more negative = better), so ascending `ORDER BY rank` puts best
first. `LEFT JOIN chats` supplies the tree-grouping key (`root_id`) + the
breadcrumb title.

**Deep-link resolver** (pure, in the repo or the route):
- `message` → `/chat/{chatId}#m={refId}`  (reuses `handleHashScroll`)
- `chat`    → `/chat/{refId}`
- `lab`     → `/lab/{refId}`              (route exists)
- `quiz_question` → `/quiz/{quizId}`      (route exists)

## Tests — Vitest (in-memory driver), `src/lib/db/repositories/search.test.ts` (new)

Mirror `self-check.test.ts` / `repositories.test.ts` (`bootstrapWithDriver(await
createMemoryDriver())` per `beforeEach`). The FTS5 gate test runs first; the rest
`it.skipIf` on `!(await fts5Available())` so the suite is green even if sql.js
lacks FTS5 (defensive — expected to pass).

1. **FTS5 gate** — `CREATE VIRTUAL TABLE _p USING fts5(x)` + insert + `MATCH`
   works (asserts `fts5Available()` is true here; documents the assumption).
2. **Trigger sync** — insert a message → immediately `search()` finds it; update
   its content → the new term matches, the old doesn't; delete it → gone.
3. **Cascade consistency** — after `chatsRepo.deleteBranch(id)` (UX2) the
   messages/labs/quizzes in that subtree are **absent** from search results
   (their DELETE triggers fired). Sibling/parent rows remain searchable.
4. **Backfill** — pre-seed rows (chat + messages + lab + quiz) *before* the FTS
   migration is applied by booting with a driver whose schema lacks `search_fts`,
   then running the migration → `search()` returns all four kinds.
5. **`rebuildIndex()` idempotent + strips noise** — call twice; results stable;
   a message containing ```` ```mermaid ``` ```` and `$x^2$` yields a snippet
   without those tokens after rebuild (but *with* them before, per Correction 3).
6. **bm25 ranking** — two messages, one a closer lexical match, rank the closer
   one above the worse one (`hits[0].rank < hits[1].rank`).
7. **`stripIndexNoise` (pure)** — strips mermaid + KaTeX; keeps inline code and
   fenced non-mermaid code.
8. **`buildMatchQuery` (pure)** — `"foo" "bar"` for `foo bar`; empty → null;
   specials (`"*`, `"`, `:`, `(`) inside a token are quoted (no exception).
9. **`renderSnippet` (pure)** — splits on `\x01`/`\x02` into marked/unmarked
   segments; handles a snippet with no markers and one with multiple.
10. **Deep-link resolver** — correct href per kind (incl. `quiz_question` →
    `/quiz/{quizId}`).

`pnpm check` + `pnpm lint` clean. (Desktop FTS5 is manual-only — not CI.)

## Risks / edge cases

- **FTS5 absent in one runtime.** The hard gate catches it; `fts5Available()`
  degrades the UI gracefully (notice instead of a broken search). Decision I
  resolves the desktop case at gate time.
- **`DELETE FROM search_fts WHERE kind=? AND ref_id=?` scans.** UNINDEXED-column
  filtering is a linear scan over the FTS doclist. At single-learner scale
  (thousands of rows) this is sub-millisecond; only matters if the corpus ever
  gets huge (defer → then switch the delete triggers to rowid-keyed deletes with
  a side table, or move to external-content).
- **`chats` UPDATE on every `touch()`.** Re-indexes the title row on each
  `updatedAt` bump (chat store touches on activity). Negligible; if it ever
  shows up in profiles, narrow the `UPDATE` trigger to fire only when
  `new.title IS OLD.title` is false (guard). Left simple for v1.
- **Snippet marker collisions.** `\x01`/`\x02` cannot occur in Markdown text; if
  a pathological payload inserted them, `renderSnippet` would mis-mark. Acceptable
  (defensive: `stripIndexNoise`/sanitize could scrub, but not worth it).
- **Migration on a large pre-existing DB.** The backfill is four `INSERT…SELECT`
  statements in one transaction (OPFS worker / Tauri batch each wrap in BEGIN).
  For a very large DB this is a one-time multi-second cost at upgrade; acceptable
  and offline. `rebuildIndex()` chunks its inserts to avoid a giant single batch.
- **`rebuildIndex()` concurrent with a live write.** Single-user + synchronous
  app logic → not a concern; if async streaming ever interleaves, a duplicate
  FTS row could appear (self-heals on next rebuild). Documented as benign.
- **MATCH syntax errors from user input.** `buildMatchQuery` double-quotes every
  token, neutralising FTS5 operators; a malformed query still can't throw (quoted
  strings are literal). If `search()` ever throws, the route shows the error line
  (no crash).

## Out of scope

- Semantic / vector search (UX5 — gated, separate decision doc).
- `branch_sources` indexing (covered transitively by the source message).
- Prefix matching (`token*`) and `trigram` tokenizer (deferred).
- `LIKE` fallback (only if the hard gate forces it — Decision I at gate time).
- Cross-tree relevance tuning / faceted filters / search history.
