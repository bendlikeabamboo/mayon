# v0.1.0 Release, Versioning, and Dev/Prod Container Split

## Context & goals

Ship the first stable release `v0.1.0` at current HEAD (`d65f9cb`) and fix the release
machinery + dev/prod topology so that:

1. The release is a hard contract: git tag `vX.Y.Z` ⟺ all three `package.json` versions ⟺
   a `CHANGELOG.md` `[X.Y.Z]` section. CI fails the release otherwise.
2. GHCR publishes **both** images (web SPA + server), since the app is server-required;
   `docker compose up` works from pure pulls.
3. Dev and prod can run **simultaneously** with no shared mutable state: prod is a
   stable daily-driver (prebuilt images, real data), dev is hot-reload (host tsx server
   + Vite) against an **isolated** database.

### Resolved decisions (from planning interview)
- **Re-tag `v0.1.0` at HEAD.** The existing `v0.1.0` points at `162ca0f` (4 commits behind);
  it will be force-moved to `d65f9cb` and force-pushed; the `:0.1.0` GHCR image is overwritten.
- **Two GHCR images**, prod `docker-compose.yml` consumes prebuilt `image:` tags.
- **Version guard:** tag must equal all three `package.json` versions AND a CHANGELOG
  `[X.Y.Z]` section must exist.
- **Dev is fully Dockerized** for dev/prod parity. The dev server runs in a container
  (`tsx watch`), so `DATABASE_URL` is always `…@db:5432/mayon` via compose `environment:`
  in both dev and prod — no host `.env`, `tsx --env-file`, or host db port needed. The dev
  stack is a separate compose project (`mayon-dev`) with disjoint volumes/ports.

### Topology target
| | Prod (daily-driver) | Dev |
|---|---|---|
| Web | `ghcr.io/.../mayon:0.1.0`, host `:8080` | Vite HMR (in container), host `:5173` |
| Server | `ghcr.io/.../mayon-server:0.1.0`, docker-internal `4319` | `tsx watch` (in container), docker-internal `4319` |
| DB | `postgres:17`, volume `pg-data`, **no host port** | `postgres:17`, volume `pg-data-dev`, **no host port** |
| Compose project | `mayon` (default) | `mayon-dev` |
| `DATABASE_URL` (server) | `…@db:5432/mayon` (compose `environment:`) | `…@db:5432/mayon` (compose `environment:`) |

No host-port collisions across the two stacks, so both run at once.

### 3. Split prod `docker-compose.yml` to use prebuilt images

- `web`: `image: ghcr.io/bendlikeabamboo/mayon:${MAYON_VERSION:-latest}`, `ports: 8080:80`,

- End-user flow becomes `docker compose pull && docker compose up -d`. Upgrade flow:

### 4. All-Docker dev stack — `docker-compose.dev.yml` + dev Dockerfiles
New file, full dev/prod parity: web (Vite HMR) + server (`tsx watch`) + db, all in
containers, separate project/volume/port from prod:
- `name: mayon-dev` keeps networks/volumes/disks disjoint from prod.
- `web`: builds `Dockerfile.dev` (node + deps + built `@mayon/shared`), `ports: 5173:5173`,
  binds `./src` and `./static` for HMR (node_modules + shared dist from image),
  `vite.config.ts` gets `server.host: true` so the in-container dev server is reachable.
- `server`: builds `server/Dockerfile.dev` (node + deps + built `@mayon/shared`), `expose: 4319`,
  binds `./server/src` (HMR), `./src/lib/db/schema.ts:ro` (`@mayon/schema` alias target),
  `./drizzle:ro` + `MIGRATIONS_DIR=/app/drizzle` (dev cwd is `/app/server`), volume
  `server-data-dev`, `environment: DATABASE_URL=…@db:5432/mayon`, CMD `pnpm --filter @mayon/server dev`.
- `db`: `postgres:17-alpine`, volume `pg-data-dev`, **no host port** (server reaches it
  as `db:5432` on the internal network, same as prod), same `POSTGRES_*` env + healthcheck.
- `Dockerfile.dev` uses `--filter mayon...` (excludes server native deps like better-sqlite3);
  `server/Dockerfile.dev` uses `--filter @mayon/server...`. Rebuild via `pnpm dev:build`.

### 7. Release workflow — version guard + dual image publish

## Risks & caveats
- **Force-moving `v0.1.0`** rewrites a public tag; anyone who pinned `v0.1.0`@162ca0f gets a
  different image after the re-publish. Pre-1.0, acceptable per decision.
- **Dev image rebuild on dep/config/`@mayon/shared` changes:** source HMR covers `src/`,
  `static`, and `server/src`, but changing `package.json`/lockfile, vite/svelte config, or
  the shared workspace package requires `pnpm dev:build`. Documented in the dev script help.
- **`pnpm deploy --legacy`** in `server/Dockerfile` must still produce a working server image
  once published; validate the published `:0.1.0` server image actually boots and serves
  `/api/health` (migrations run, `pg` cap present).
- Older stray tags (`0.0.1` w/o `v`, `v0.0.x`) are **left in place** (history) — out of scope.

## Validation
- `pnpm lint && pnpm check && pnpm test` (root) and `pnpm --filter @mayon/server test` green.
- **Dev/prod concurrency:** with prod up (`docker compose up`, web `:8080`) start dev
  (`pnpm dev`) → Vite at `:5173` HMRs, dev server hot-reloads, and a write in dev does
  **not** appear in prod (different volume + project).
- **Release guard:** tag `v0.1.0` while any package.json ≠ `0.1.0` or CHANGELOG lacks
  `[0.1.0]` → `verify-version` job fails (test locally by temporarily mismatching).
- **Self-host:** fresh clone → `docker compose pull && docker compose up -d` → `:8080`
  reaches the SPA, server logs `pg: ready`, header badge shows DB ready (pg).
- **Images:** both `:0.1.0` and `:latest` present on GHCR for web and server.

## Out of scope
- Deleting/renaming historical `0.0.1`/`v0.0.x` tags.
- Major/minor (`0.1`) floating tags (kept to `{{version}}` + `latest` for v0.x simplicity).
- Changesets/release-please automation (manual tag flow retained for now).
